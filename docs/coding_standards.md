# Analysis-01 코딩 표준 및 작업 철학

본 문서는 프로젝트 전체에 적용되는 핵심 개발 철학과 코딩 스탠다드를 정의합니다. 모든 코드 작성 및 리팩토링은 이 원칙을 준수해야 합니다.

## 1. Clean Code 및 SOLID 원칙
- **가독성 최우선**: 함수/변수명은 의도를 명확히 드러내야 하며, 코드가 무엇을 하는지 설명하는 불필요한 주석은 제거합니다.
- **단일 책임 원칙 (SRP)**: 복잡한 비즈니스 통계 로직은 UI 렌더링 및 상태 관리 영역과 철저히 분리하여 `utils`나 훅으로 추출합니다.
- **Early Return**: 중첩된 `if-else` 블록을 피하고, 예외 조건은 함수 상단에서 조기 반환(`return`)합니다.
- **방어적 프로그래밍**: 외부 데이터나 전역 상태 의존 시 `?.` (Optional Chaining) 및 `|| []` (Fallback) 등을 적극 활용하여 런타임 크래시를 원천 방지합니다.

## 2. 테스트 주도 개발 (TDD)
- 비즈니스 로직(예: 요인 추출 로직, 통계 행렬 정렬)은 UI에 종속되지 않는 순수 함수로 작성하고 단위 테스트(Unit Test) 작성의 기반을 마련합니다.
- 테스트는 RED(실패) - GREEN(통과) - REFACTOR(개선) 사이클을 지향합니다.

## 3. Frontend (React/Zustand) 제약 사항
- **거대 컴포넌트 금지**: 파일 라인 수가 길어지면 아토믹(Atomic) 디자인 원칙에 따라 서브 컴포넌트로 강제 분할합니다.
- **에러 바운더리 적용**: UI의 특정 부분이 실패하더라도 화면 전체의 `White Screen of Death`로 번지지 않도록 에러 바운더리를 씌웁니다.
- **상태 직접 변이 금지**: 상태 업데이트는 반드시 불변성(Immutability)을 유지해야 합니다.

## 4. 작업 철학 (PDCA & 머스크의 ATOM)
항상 아래의 5단계를 순서대로 거치며 기술 부채를 통제합니다.

1. **요구사항을 의심하라 (Question)**: 모든 요구사항의 본질을 파악하고 진짜 필요한 것인지 묻습니다.
2. **삭제하라 (Delete)**: 쓰이지 않는 파일, 주석, 오버엔지니어링된 로직은 가차없이 지웁니다.
3. **단순화하라 (Simplify)**: 삭제 후 남은 핵심만을 간결하게 최적화합니다.
4. **가속하라 (Accelerate)**: 1~3단계를 통과한 작업에 한해 속도를 높입니다.
5. **자동화하라 (Automate)**: 안정되고 확정된 로직에 대해서만 테스트 자동화 등을 적용합니다.
